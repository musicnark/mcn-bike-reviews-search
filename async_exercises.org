* Asynchronous Parallelism Exercises with `core.async`

These exercises are designed to help you practice and solidify your understanding of `core.async` for parallelism workflows in Clojure. They build on the concepts we've discussed, such as non-blocking I/O, `go` blocks, and channels.

---

**Note:** For these exercises, assume you have a list of URLs available, similar to how you'd get them from your `input-table` and `parse-urls` functions. You might want to define a small sample list for testing, e.g.:

```clojure
(def sample-urls
  ["https://www.motorcyclenews.com/bike-reviews/kawasaki/kle500/2026/"
   "https://www.motorcyclenews.com/bike-reviews/yamaha/mt-07/2020/"
   "https://www.motorcyclenews.com/bike-reviews/honda/cbr600rr/2024/"])
```

---

*   **Exercise 1: The Collector**

    *   **Goal:** Fetch the first 20 URLs from your CSV file. Collect all the **successfully parsed** bike data maps into a single Clojure vector. At the end, print the total count of bikes you successfully retrieved. Ignore any requests that failed to fetch or parse.

    *   **Concepts you'll practice:**
        *   **Fan-out/Fan-in:** Launching many concurrent operations and collecting all the results.
        *   **Filtering:** Processing a stream of results from a channel and only keeping the ones you want.

    *   **Hints:**
        1.  Start with the `map` and `merge` pattern from our last example to launch all the requests and get a single channel of results.
        2.  Use a `go-loop` or similar construct to read from your merged channel.
        3.  Inside the loop, check each result. Is it an `ok?` result? Has it been parsed correctly?
        4.  Instead of just printing, accumulate the successful results into a vector. When the loop is done, you'll have your final collection.
        5.  Consider using `clojure.core.async/into`, which can simplify collecting all items from a channel into a collection.

*   **Exercise 2: The Rate-Limiter**

    *   **Goal:** Your scraper is too aggressive! The server is complaining. Modify your process to fetch all the URLs from the CSV, but ensure that **no more than 8 HTTP requests are active at any given time.**

    *   **Concepts you'll practice:**
        *   **Controlling Parallelism:** This is a crucial real-world skill for any kind of batch processing.
        *   **Pipelines:** A powerful abstraction for creating concurrent "assembly lines."

    *   **Hints:**
        1.  You'll want to use `clojure.core.async/pipeline`. You'll need to add it to your `:refer` list for `clojure.core.async`.
        2.  A `pipeline` connects an input channel (where you'll put the URLs) to an output channel (where the results will appear).
        3.  The key argument to `pipeline` is the parallelism number, `n`. This number tells `core.async` to ensure that no more than `n` of your functions are running at once.
        4.  Your workflow will look something like this:
            *   Create an `in-chan` for URLs and an `out-chan` for results.
            *   Define a transformation function that takes one URL and does the work (fetches and parses).
            *   Set up the `pipeline` with a parallelism of `8`, connecting your channels and the function.
            *   Create a `go` block to feed all your URLs into the `in-chan`, then `close!` it.
            *   Create another loop to read all the results from the `out-chan` until it's empty and closed.

*   **Exercise 3: The Impatient Fetcher**

    *   **Goal:** A single request to a specific URL seems to be hanging sometimes, blocking your whole process. Write a function that tries to fetch and parse a single URL, but **gives up if it takes longer than 4 seconds.** The function should return the parsed bike map on success, or a specific `:timeout` error map on failure.

    *   **Concepts you'll practice:**
        *   **Handling Timeouts:** A fundamental requirement for robust network programming.
        *   **Racing Channels:** Using `alts!` to act on the first of several possible events.

    *   **Hints:**
        1.  The star of this show is `clojure.core.async/alts!`. You'll need to add it to your `:refer` list for `clojure.core.async`.
        2.  You'll also need `clojure.core.async/timeout`, which creates a channel that closes after a specified number of milliseconds.
        3.  The pattern looks like this:
            *   Get the `response-chan` by calling `fetch-bikes-async`.
            *   Create a `timeout-chan` with `(async/timeout 4000)`.
            *   Inside a `go` block, use `let [[value channel] (alts! [response-chan timeout-chan])]`.
            *   Check which `channel` won the race. If it was `response-chan`, you have a `value` to process. If it was `timeout-chan`, then you should return your custom timeout error.
